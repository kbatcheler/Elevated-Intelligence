# Phase 3: UI surfacing the verified vs modelled split

Scope: render the Phase 2 data so a buyer can see at a glance how grounded each layer's diagnosis is. Source URL pills next to verified claims, confidence bands next to modelled claims, provenance tooltip per claim, per-layer count surfacing in the header. The credibility close happens when a buyer clicks a pill and sees the actual press release that supports the claim. That moment is what Phase 3 ships.

This is Phase 3 of four. Phase 4 will add operational tooling and per-claim interactions (refresh layer, edit claim, time-travel).

Five sub-phases below. Each has acceptance criteria. Typecheck must pass between sub-phases. Run end to end. Stop only on a hard blocker or a destructive ambiguity.

## Architectural decisions, pre-resolved (do not re-ask)

- Annotations attach to STRUCTURED FIELDS, not arbitrary substrings. The `claim_path` field (e.g. `causes[0].title`, `actions[2].impact`) resolves to a complete field value. The whole field is the claim. No fuzzy text matching, no NLP, no parser.
- Verified pills are gold, brand-aligned, clickable, open source URL in new tab.
- Modelled bands are cream/gold-soft/coral by confidence tier. Display the confidence percentage and a one-word basis tag. Hover shows full basis text.
- Unsupported claims (no entry in either array) render as plain text. No "unsupported" badge. Silent.
- Layer header carries the per-layer counts (X verified, Y modelled) as the system's grounding signal at a glance.
- No "Challenge this" or "Edit claim" affordances in Phase 3. Those are Phase 4.
- All pill/band components must degrade gracefully at viewport widths down to 1024px (existing min supported). Mobile splash below 1024px stays unchanged.
- Existing `data/layers.ts` schema file unchanged. Phase 3 only touches rendering, not data shape.

## Sub-phase 3.1: Component primitives

Add four new components in `artifacts/portal/src/components/claims/`.

### `VerifiedPill.tsx`

Props:
```typescript
{
  sourceUrls: string[];
  sourceTitles?: string[];
  verifiedBy: string;       // "gemini-2.5-pro" or "anthropic-web-search"
  verifiedAt: string;       // ISO timestamp
}
```

Behaviour:
- Renders a single small pill: `[icon] stripe.com` (extract domain from first URL).
- Background: gold (`--gold`). Text: navy. Border-radius: 9999px. Padding: 2px 8px. Font: 10px semibold.
- If more than one URL, show domain of first + "+N more" badge.
- On hover (after 300ms delay), expands a tooltip card below showing:
  - "VERIFIED" eyebrow in small caps
  - Each source URL as a clickable link (opens new tab) with the page title if available
  - "Verified by [model], [relative time]" footer
- On click of the pill itself, opens first URL in new tab.
- Keyboard accessible: focusable, Enter opens first URL, Tab cycles through nested URL links in expanded tooltip.

### `ModelledBand.tsx`

Props:
```typescript
{
  confidence: number;       // 0-100
  basis: string;            // e.g. "sector benchmark", "industry pattern"
  inferredFrom?: string[];  // optional list of observations the inference drew from
}
```

Behaviour:
- Renders a small pill: `67% · MODELLED` or similar.
- Background colour by confidence tier:
  - 75-95%: `--cream`
  - 50-74%: `--gold-soft`
  - 30-49%: `--coral`
  - Below 30%: should not appear in production data; if it does, render in coral with a console warning.
- Text: navy in all cases. Same dimensions as VerifiedPill.
- The basis tag appears in a small chip beside the percentage: `67% · sector benchmark`.
- On hover, tooltip shows the full basis text plus the inferred-from observations if present.
- Not clickable. No link target. This is intentional, modelled claims do not have a source to click through to.

### `ProvenanceTooltip.tsx`

Shared tooltip primitive used by both VerifiedPill and ModelledBand. Uses Radix UI tooltip (or whatever the project uses for popovers; consistent with existing tooltips in the codebase).

- Slight delay on open (300ms).
- Closes immediately on mouse leave or Escape.
- Positioned `bottom`, falls back to `top` if near viewport edge.
- Background: navy. Text: cream. Border: 1px gold-soft. Max-width: 320px.

### `ClaimAnnotation.tsx`

Wrapper component that looks up a claim by path and renders the right primitive.

Props:
```typescript
{
  claimPath: string;        // e.g. "causes[0].title"
  verifiedClaims: VerifiedClaim[];
  modelledClaims: ModelledClaim[];
  align?: "inline" | "trailing" | "leading";  // default "trailing"
}
```

Behaviour:
- Search `verifiedClaims` first for matching `claim_path`. If found, render `<VerifiedPill>` with that claim's data.
- Else search `modelledClaims`. If found, render `<ModelledBand>`.
- Else render nothing (the unsupported case).
- `align="trailing"` (default) renders the pill after the field's content with a small left margin.
- `align="leading"` renders before with right margin.
- `align="inline"` is a flex-inline pattern, useful in card layouts.

### Acceptance for 3.1

1. Storybook (or whatever the existing component preview pattern is) shows all three pill variants: VerifiedPill with 1 URL, VerifiedPill with 3 URLs, ModelledBand at three confidence tiers (high/medium/low).
2. Hover behaviour works in the preview: tooltips expand, show correct content, dismiss correctly.
3. Click on VerifiedPill opens the URL in a new tab.
4. Components are typed against the existing `VerifiedClaim` and `ModelledClaim` types from Phase 2.
5. Typecheck passes.

| §4 Proof | Each proof item source | `proof.items[N].source` |
| §4 Proof | Each gap description | `gaps[N].description` |

### Implementation pattern

Around each annotated field, wrap with the annotation. Example for `causes[0].title`:

```tsx
<h4 className="cause-title">
  {cause.title}
  <ClaimAnnotation
    claimPath={`causes[${idx}].title`}
    verifiedClaims={layer.verified_claims}
    modelledClaims={layer.modelled_claims}
    align="trailing"
  />
</h4>
```

For block-level fields like `narrative` (paragraph) where a trailing pill would orphan visually, use `align="trailing"` but place the annotation at the END of the paragraph in its own line.

### Layout adjustments

The existing layer page layouts assume tight prose. Pills add ~24px of horizontal space per occurrence. Audit each section for layout overflow at viewport 1280px. Specifically:

- Cause cards: ensure the cause title row wraps gracefully when a pill appends.
- Action cards: same.
- Metric tiles: pills here will be tight; use `align="inline"` and place in the tile footer.
- Hypothesis blocks: pills can wrap to next line, this is fine.

## Sub-phase 3.3: Layer header count surface

The buyer's at-a-glance grounding signal lives in the layer header, alongside the existing confidence band and dual signal.

### Header strip additions

In the layer header section (between the layer title/subtitle and the first §1 card), add a single row of count badges:

```
[12 VERIFIED]   [8 MODELLED]   [23 SOURCES]
```

Style:
- Each badge is a small pill, 24px tall, padded.
- VERIFIED badge: gold background, navy text, count in serif bold.
- MODELLED badge: cream background, navy text.
- SOURCES badge: navy background, cream text, count of unique URLs across verified claims.
- Each badge is hoverable, shows a tooltip explaining the count.

### Tooltip content

- VERIFIED: "Claims backed by at least one source URL. The system found public evidence supporting these statements."
- MODELLED: "Claims that draw on sector benchmarks and public signals. The system marks these explicitly so you know which parts of the diagnosis are hypothesis."
- SOURCES: "Distinct source URLs cited across this layer's verified claims. Click any verified claim's pill to open its source."

### Cross-page consistency

The same count strip appears on the Morning Brief layer cards (existing top findings cards in `MorningBrief.tsx`). Smaller variant: just the two numbers without the SOURCES count, as a compact `12·8` badge in the card corner.

The cross-layer map page (Cross-layer dependency graph) does NOT add per-node counts in Phase 3. Keep the existing visualization. Phase 4 might revisit.

## Sub-phase 3.4: Provenance polish and dead-link handling

Real-world sources break. Pages get redirected, taken down, or paywalled. Phase 3 must handle this gracefully.

### Dead-link checking

On layer page load, the frontend does NOT pre-check URLs (would be slow and add traffic). Instead:

- Each VerifiedPill shows the domain. The pill is clickable. If the URL is dead or paywalled, that's the buyer's problem to discover on click. Not a Phase 3 concern.
- BUT, when a buyer clicks through to a 404 or paywall, that's a credibility moment lost. Add a small "Report broken link" link in the tooltip footer. On click, sends a POST to `/api/claims/report-broken` with the tenant id, layer key, claim_path, source_url. Backend logs it. No UI changes after report; just a small "Thanks" confirmation that fades.

### Backend endpoint

`POST /api/claims/report-broken`:
- Body: `{tenant_id, layer_key, claim_path, source_url}`
- Inserts into a new lightweight table `claim_broken_reports` (id, all body fields, reported_at, reported_by). Add the migration.
- Returns 200 ok.
- This is reporting only; Phase 3 does not act on the data. Phase 4 might surface a "X broken links flagged" admin view.

### Provenance tooltip polish

For VerifiedPill tooltips, surface:
- The page title (from `sourceTitles[i]` if present, else "View source")
- The domain in slate text below the title
- The "Verified by [model], [relative time]" footer in even smaller slate
- A divider, then the "Report broken link" affordance

For ModelledBand tooltips:
- The basis text in full
- A "Why modelled?" expandable that, when clicked, shows:
  - "This claim was generated from public signals and sector knowledge rather than a specific verifiable source. The system marks it explicitly so you can weigh it appropriately. Phase 2 of the pipeline (the Challenge stage) either found no public source for this claim, or the available evidence was indirect."
- This explainer copy is reusable across all modelled tooltips.

### Acceptance for 3.4

1. Migration adds `claim_broken_reports` table.
2. `POST /api/claims/report-broken` accepts the report and returns 200.
3. Clicking "Report broken link" in a tooltip fires the request, shows "Thanks" confirmation.
4. ModelledBand tooltip shows the "Why modelled?" expandable with the reusable explainer.
5. Typecheck passes.

## Decisions pre-resolved (do not re-ask)

- Field-level annotations, never sub-string parsing
- VerifiedPill is clickable, ModelledBand is not
- Three confidence tiers for ModelledBand (high/medium/low), nothing finer
- No "unsupported" badge for unmarked claims
- Header strip is three badges (verified, modelled, sources), in that order
- 300ms hover delay on tooltips, consistent across the app
- Broken-link reports go to a new lightweight table, no immediate UI surfacing
- 1280px is the layout-test viewport; mobile splash below 1024px stays as-is
